# DTRAP ML Research: Cyber Threat Prediction with Machine Learning for Linux Systems

Code and data accompanying the manuscript *"Evaluating Cyber Threat
Prediction with Machine Learning for Linux Systems"* (submitted to ACM
*Digital Threats: Research and Practice*, DTRAP-2025-0106).

This repository reproduces every quantitative result reported in the paper:
data generation, feature engineering, model training/evaluation for four
classifiers (Linear SVM, Random Forest, Gradient Boosting, a DNN), a
non-circular rule-based detection baseline, a cross-validation variance
check, and the figures built from those results. `REPRO_NOTES.md` is the
canonical provenance record explaining how every number in the paper was
produced, including bugs found and fixed along the way (label leakage,
preprocessing leakage, a run-identifier shortcut) — read it first if you want
the full story, not just the commands.

## Dataset

`data/auditbeat_dataset.parquet` contains 226,760 raw Auditbeat events
collected from a single-host Linux lab environment running [Atomic Red
Team](https://github.com/redcanaryco/atomic-red-team)-style simulations of
five MITRE ATT&CK techniques (T1087, T1082, T1166, T1169, T1016), audited via
the rules in `linux_audit.rules`. After filtering, this yields the 86,689
labelled events (8 classes) used in the paper.

The original collection host's real username and hostname have been
removed from this file (string-replaced with generic placeholders across all
39 affected columns) before publication. `redact_pii.py` documents exactly
how, and `REPRO_NOTES.md` §8 confirms this redaction does not change any
reported metric for the three deterministic classical models or the
rule-based baseline. That comparison predates the reproducibility fix in
§9.1 (the DNN was unseeded at the time and showed run-to-run noise
unrelated to redaction); the DNN is now fully deterministic, and the
redaction comparison should be re-run against the seeded scripts if it needs
to be cited again.

```
python -m venv .venv
source .venv/bin/activate          # or .venv\Scripts\activate on Windows

```

Tested with Python 3.11.

## Reproducing the results

All commands are run from the `repro/` directory:

```
cd repro
python build_features.py          # feature engineering -> repro/artifacts/
python train_models.py             # Protocol A (57 features) -> repro/results/metrics.json
python train_models_ablation.py    # Protocol B (drops 5 run-identifier columns) -> repro/results_ablation/metrics.json
python rule_baseline.py             # rule-based ATT&CK baseline -> repro/results_rule_baseline/metrics.json
python cv_repeated.py               # stratified 5-fold CV (RF, GB) -> repro/results_cv/cv_metrics.json
python make_figures.py              # regenerates all figures -> repro/figures/*.png
```

`repro/artifacts/` (intermediate feature matrices) is generated by
`build_features.py` and is not checked in; everything downstream of it is
regenerated from the dataset in `data/`.

Saved outputs from the run reported in the paper are already included under
`repro/results*/` and `repro/figures/`, so you can inspect the headline
numbers without re-running anything.

## Headline results (Protocol A, time-based 70/30 split, n=26,007 test events)

These numbers are now fully deterministic: every script (`build_features.py`,
`train_models.py`, `train_models_ablation.py`) seeds NumPy, Python's `random`,
and TensorFlow/Keras (`tf.keras.utils.set_random_seed`), and pins
`TF_DETERMINISTIC_OPS`/`TF_CUDNN_DETERMINISTIC`. Re-running the pipeline from
a clean checkout reproduces every value below bit-for-bit (verified by
running each script twice and diffing the saved `metrics.json`).

| Model | Accuracy | Macro-F1 |
|---|---|---|
| Random Forest | 99.98% | 99.99% |
| Gradient Boosting | 98.82% | 72.61% |
| Linear SVM | 97.13% | 74.62% |
| DNN | 96.75% | 73.66% |
| Rule-based baseline | 91.25% | 77.13% |

See `REPRO_NOTES.md` for Protocol B (run-identifier ablation), the
cross-validation variance check, and full methodology. Note: an earlier
version of this study reported a DNN-specific "run-identifier shortcut"
effect (Protocol A macro-F1 61.05% -> Protocol B 80.12%) based on a single
unseeded run pair. With seeding fixed, the deterministic result is Protocol A
73.66% -> Protocol B 72.58% (a -1.08 point change) - i.e. no shortcut effect.
`REPRO_NOTES.md` §9.1 has the full account, including a 5-seed variance check
that was run before the fix to confirm the original finding was noise.

## Repository layout

```
data/auditbeat_dataset.parquet   Redacted raw Auditbeat dataset (86,689 events after filtering)
linux_audit.rules                 Linux Audit Framework rules used for data collection
simulator.sh                      Attack-simulation script (Atomic-Red-Team-style)
redact_pii.py                     Script used to redact the original dataset before publication
repro/
  build_features.py               Feature engineering (Protocol A, 57 features)
  train_models.py                  Trains SVM, Random Forest, Gradient Boosting, DNN (Protocol A)
  train_models_ablation.py         Protocol B (drops 5 run/session-identifier columns)
  rule_baseline.py                 Non-circular rule-based ATT&CK detection baseline
  cv_repeated.py                   Stratified 5-fold CV (variance estimate, RF/GB)
  make_figures.py                  Regenerates all figures from saved results
  results/, results_ablation/, results_rule_baseline/, results_cv/
                                    Saved metrics.json, confusion matrices, DNN training history
  figures/                         All figures used in the manuscript (PNG)
REPRO_NOTES.md                    Full provenance record for every number in the paper
```
